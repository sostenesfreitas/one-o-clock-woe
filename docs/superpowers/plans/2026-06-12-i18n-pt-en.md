# i18n pt-BR / English — Implementation Plan (Fase 0 + Fase 1)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add a per-device `PT | EN` language toggle to the single-file WoE app, defaulting to pt-BR, and translate the core screens (nav, League, Overrun, Roster, Auction GL/Overrun) out of Thai.

**Architecture:** A central `LOCALES` dictionary + a `t('key', params)` lookup with `locale → pt-BR → key` fallback. Static `<body>` markup is annotated with `data-i18n` / `data-i18n-attr` and filled by `applyStaticI18n()`. The active locale lives in a module-level `currentLocale` (OUTSIDE `state`, persisted to `localStorage`), and `setLocale()` re-applies static markup + calls `render()`. Phase 0 builds the engine; Phase 1 converts the core screens one at a time.

**Tech Stack:** Vanilla JS (no build, no modules), single `<script>` in `app.html`. Dependency-free Node test harness (`test/run.js` + `test/harness.js`). Firebase compat SDK (untouched here).

**Spec:** `docs/superpowers/specs/2026-06-12-i18n-language-toggle-design.md`

**Branch:** `feat/i18n-pt-en` (already created; spec already committed there).

```js
});

  eq(app.call("t", "common.save"), "Save");

  eq(app.call("t", "__definitely_missing__"), "__definitely_missing__");

  app.call("setLocale", "pt-BR");
```

```js
/* ===== i18n — pt-BR / English (see docs/superpowers/specs/2026-06-12-i18n-language-toggle-design.md) =====
 * Central dictionary + t() lookup. Game jargon (Card, Knight, party, item
 * names) stays literal in English and is NOT keyed here. currentLocale lives
 * OUTSIDE `state` so it never syncs to Firebase and save() never serializes it. */
const LOCALES = {
  "pt-BR": {
    "common.save": "Salvar",
    "queue.badge": "Fila #{n}",
  },
  "en": {
    "common.save": "Save",
    "queue.badge": "Queue #{n}",
  },
};
let currentLocale = (function () {
  try {
    const v = localStorage.getItem("woe_locale");
    return (v && LOCALES[v]) ? v : "pt-BR";
  } catch (e) { return "pt-BR"; }
})();

function t(key, params) {
  const dict = LOCALES[currentLocale] || LOCALES["pt-BR"];
  let s = dict[key];
  if (s == null) s = LOCALES["pt-BR"][key];
  if (s == null) s = key;
  if (params) {
    for (const k in params) s = s.split("{" + k + "}").join(String(params[k]));
  }
  return s;
}
```

```js
/* Fill static markup: <el data-i18n="key"> sets textContent;
 * <el data-i18n-attr="placeholder:key,title:key2"> sets attributes. */
function applyStaticI18n(root) {
  root = root || document;
  root.querySelectorAll("[data-i18n]").forEach(function (el) {
    el.textContent = t(el.getAttribute("data-i18n"));
  });
  root.querySelectorAll("[data-i18n-attr]").forEach(function (el) {
    el.getAttribute("data-i18n-attr").split(",").forEach(function (pair) {
      const i = pair.indexOf(":");
      if (i < 0) return;
      const attr = pair.slice(0, i).trim();
      const key = pair.slice(i + 1).trim();
      if (attr && key) el.setAttribute(attr, t(key));
    });
  });
}

function setLocale(loc) {
  if (!LOCALES[loc]) return;            // invalid → no-op
  currentLocale = loc;
  try { localStorage.setItem("woe_locale", loc); } catch (e) {}
  applyStaticI18n(document);
  if (typeof render === "function") render();
  if (typeof updateLocaleSwitcherUI === "function") updateLocaleSwitcherUI();
}
```

> `updateLocaleSwitcherUI` is defined in Task 3; the `typeof` guard keeps `setLocale` callable in the harness (where the switcher DOM may be absent) and before Task 3 lands.

## PHASE 1 — Core screens

Convert nav + League + Overrun + Roster + Auction GL/Overrun from Thai to keyed `t()` / `data-i18n`, populating both `LOCALES["pt-BR"]` and `LOCALES["en"]`. **Each screen is one task** following the identical procedure below. The string translations themselves are produced during implementation and collected in `docs/i18n-glossary.md` for user review (game jargon stays literal English — not keyed).

### Per-screen procedure (apply to every Phase 1 task)

1. **Find the Thai** for the screen: `grep -nP '[\x{0E00}-\x{0E7F}]' app.html` (Thai Unicode block), narrowed to that screen's render function / markup line range.
2. For each Thai UI string (NOT game jargon), **add a key** to both locales in `LOCALES`. Naming: `area.item` (e.g. `nav.league`, `roster.add`, `auction.shortage`). Reuse `common.*` for repeated verbs (Salvar/Cancelar/Excluir/Confirmar).
3. **Replace the literal**: dynamic strings in render functions → `t('key')` (or `t('key',{...})` for interpolation); static markup → `data-i18n="key"` / `data-i18n-attr="placeholder:key"`.
4. **Record** each term in `docs/i18n-glossary.md` (TH | EN | pt-BR | key).
5. **Update tests**: any assertion in `test/run.js` matching a now-translated Thai literal switches to either the `data-i18n` key / structural check, or `setLocale('pt-BR')` + the pt-BR text. Convert in the SAME commit as the screen.
6. **Verify**: `node test/run.js` → `0 failed`; `node test/parse-check.js` → `PARSE OK`.
7. **Commit** the screen.

- **Never pass HTML through `t()`** — it returns plain text. Keep existing escaping (e.g. `escapeHtml`, `WHEEL_SAFE_KEY_RE`) exactly where it is; translate the human-readable fragments only.
- **⚠️ `t` shadowing (code-review finding):** several existing functions declare a function-local `const t = ...` (e.g. `app.html:~2995, 3028, 3103, 3128, 3329-3348, 5062`). Inside those functions the GLOBAL i18n translator `t()` is unreachable — a bare `t('key')` there silently calls the local `t` and breaks. Before converting a string, check whether the enclosing function already binds `t`; if so, rename the local or call the translator via an alias. This is the single most likely Phase 1 footgun.
- **`data-i18n` only on truly static markup.** Elements rewritten by render functions must use `t()` inside the render, not `data-i18n` (otherwise the dynamic content gets clobbered).
- **CLAUDE.md gate is mandatory** every commit: parse check + `node test/run.js` green; `/code-review` before commit; never merge to `main` without user confirmation; behavior change ⇒ test update in the same commit.
- **Test conventions (verified):** groups = `console.log("\n[name]")`; tests = `t("desc", () => {...})`; assertions = `eq(actual, expected, msg?)` / `ok(cond, msg?)`; one shared `const app = loadApp()` near the top is reused across groups. The runner's `t` shadows nothing — the app's translator is only ever `app.call("t", ...)`.
